## Workstream 1: Duplicate Connection Fix

**Problem**: When both peers connect simultaneously, `add_peer` pushed without dedup, causing duplicate PeerInfo entries and duplicate messages.

**Solution**:
- `RegisterResult` enum (Registered/Replaced/Duplicate) in `state.rs`
- `add_peer` compares session IDs: same session = Duplicate, different = Replaced (evicts old)
- `handle_connection` checks for Duplicate → sends Close frame + returns
- Pre-connect address guard in `connect_to_peer` skips already-connected addresses
- Helper methods: `is_peer_connected_by_name`, `is_peer_connected_by_addr`

**Tests**: 3 new unit tests (fresh register, duplicate same session, replace different session)

## Workstream 2: Task Board

**Design**: Deterministic task management — no LLM needed. Tasks with status/assignee/dependencies, auto-unblock when deps complete.

**Key structures**:
- `Task` struct with id, title, description, status (Todo/InProgress/Done/Blocked), assignee, priority (Low/Medium/High/Critical), depends_on, timestamps
- Auto-unblock: 3-pass approach to satisfy borrow checker — (1) collect done IDs, (2) find blocked tasks whose deps are all done, (3) mutate

**API**: 6 endpoints (GET/POST tasks, GET/PATCH/DELETE task, POST assign)
**Wire**: 3 message types (task.assign, task.update, task.complete) with payloads
**Dashboard**: Task board with colored status badges, priority indicators, status dropdowns, add form

## Workstream 3: Audit Trail

**Design**: Append-only signed entries for every project mutation. Cryptographically verifiable by any participant.

**Signature format**: `"{timestamp_rfc3339}|{author_did}|{action}|{detail}"` signed with Ed25519, stored as base58.

**Auto-logged events**: task.created, task.updated, task.assigned, task.completed, agent.clocked_in, agent.clocked_out, project.created, project.stage_changed

**Dashboard**: Scrollable reverse-chronological audit trail with timestamp, action, detail, author columns.

## Workstream 4: Stage-Gated Workflows

**Stages**: Investigation → Implementation → Review → Integration → Deployment

**Permission matrix**: Per-stage, per-role capability sets (read/write/commit/approve/deploy). Owner/Overseer have broadest access, Observer is read-only throughout.

**Guards**: `can_advance()` checks all tasks are Done before allowing stage progression. Auto-audit entry on stage change.

**Dashboard**: 5-step progress bar with active/completed styling, "Advance" button (visible only when can_advance is true).

## Technical Notes

- All new `Project` fields use `#[serde(default)]` for backward compatibility with existing projects.json
- Borrow checker challenge in `update_task`: auto-unblock required 3-pass approach to avoid simultaneous mutable/immutable borrows
- `rand` crate not in dependencies — used `uuid::Uuid::new_v4().as_u128() % 50000` for test port generation
- `patch` import from axum::routing unnecessary — PATCH is a method on MethodRouter, not a free function
